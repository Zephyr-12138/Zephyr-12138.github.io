# 提交流程重复点击问题复盘：从防重到页面跳转时机

<br>

> 需求背景：用户点击“提交流程”后，前端请求提交接口，成功后通过 `location.href` 跳转到结果页。

## 问题现象

线上出现了间歇性后端报错，定位后发现是同一个用户在短时间内触发了多次提交请求。  
虽然第一次请求已经成功，但页面在 `location.href` 跳转前存在一个短暂窗口，用户重复点击按钮会再次触发接口，导致重复提交。

## 问题分析

核心链路如下：

1. 用户点击“提交”，发送请求 A。
2. 请求 A 返回成功，前端执行 `location.href = xxx`。
3. 在页面真正完成跳转前，当前页面仍可交互。
4. 用户再次点击“提交”，又发送请求 B。
5. 后端因重复提交或状态不一致而报错。

这次问题本质上不是“跳转失败”，而是“跳转前的可交互窗口”没有被正确保护。

## 解决方案

在提交流程期间引入全局 `loading`（含按钮禁用与页面级交互阻断）：

1. 首次点击后立即进入全局 loading 状态。
2. loading 状态下，提交按钮不可再次触发。
3. 请求成功后先保持 loading，再执行页面跳转。
4. 只有在跳转失败或请求异常时，才解除 loading 并允许重试。

这样可以从交互层面保证“单次提交只触发一次请求”。

## 结果

1. 重复点击不再触发重复请求。
2. 后端重复提交相关报错消失。
3. 提交流程的一致性和可预期性提升。

## 由此引发的页面跳转思考

这次复盘让我重新审视了“接口成功后立即跳转”这类写法：

1. `location.href` 是异步生效的，赋值语句执行完成并不代表页面已经离开。
2. 在“成功回调 -> 页面卸载”之间，前端仍处于可操作状态，必须显式做防重保护。
3. 关键动作（提交、支付、确认类）要按“事务”思维设计：  
   触发即上锁，结束再释放，而不是只依赖跳转来“自然结束”。
4. 页面跳转不应只是路由行为，更是状态切换点：要明确跳转前、中、后的交互策略。

## 可沉淀的实践

1. 所有关键提交接口默认接入前端防重（按钮禁用 + 全局 loading）。
2. 接口层增加幂等约束（如业务唯一键/幂等 token），前后端双保险。
3. 对“提交成功后跳转”的页面建立统一模板，避免同类问题重复出现。
