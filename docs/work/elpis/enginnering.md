# 基于 Webpack 5 完成工程化建设

## 工程化的目的是什么？

- 方便开发者以模块化的方式开发，无需担心变量冲突等问题
- 方便开发者在开发环境中可以更快地调试代码，改动代码无需手动刷新浏览器，而是通过热更新和热替换的方式实现“所改即所得”，提高开发效率
- 提供各种基础公共能力，方便开发者在业务开发过程中直接使用
- 将部署于生产环境中的代码进行压缩和丑化处理，更好地保护代码逻辑

我们能够看出，在开发环境与生产环境下，我们工程化的目的差异是很大的

那我们第一步要做的就是区分环境，在 package.json 中设置两个环境不同的打包启动脚本，在不同的环境中执行不同的脚本实现对应打包逻辑

但是两个环境下的打包行为又是存在一些交叉点，例如打包入口、模块解析规则、公共插件、分包策略,所以需要将一些公共配置提取出来，环境特定的配置应该是在公共配置的基础上继承而来，再添加特定于自己环境的配置

## 开发环境

前面说过，开发环境下的工程化很大一个目的是为了提高开发效率，我们可以先梳理一下我们想要的具体效果：

我们在编辑器中修改代码并保存，浏览器感知到代码变化并将修改那部分的代码重新请求回来并完成页面更新渲染

问题在于，浏览器肯定是无法凭自身感知到文件变化的，所以我们需要引入一个第三方作为文件与浏览器的中介，它能够监听文件变化并重新打包，由于写文件是一个很耗时的操作，所以此时的打包结果不应该落盘，而是以代码片段的方式存储于内存当中；另外，打包完成后它能够通知浏览器进行更新，为了减少更新成本，应该是只对改动部分进行更新，也就是模块热替换

项目中主要通过 express 以及两个中间件 webpack-dev-middleware 与 webpack-hot-middle ware 实现上述功能，前者用于监听文件变化并重新打包，将打包结果注入内存，后者用于与浏览器通信（实际上是通过 SSE 的方式建立），实现模块热替换


## 生产环境
生产环境下的打包产物是需要落地为文件的，并且为了优化网络传输性能，可以将代码进行压缩处理，并去除所有打印信息，隐藏代码逻辑
