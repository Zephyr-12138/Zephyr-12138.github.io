# 基于 NodeJs(Koa) 实现服务端内核引擎 - elpis-core

elpis-core 在该框架中的作用是将用户在各个目录中编写的逻辑加载到内存中，以运行时的形式处理对于页面以及 api 的请求

它的结构主要分为三层：

1. 接入层：router 路由分发、router-schema 路由规则、middleware 中间件
2. 业务层：controller 处理器、env 环境分发
3. 服务层：service 处理器

我们可以用一个请求的完整生命周期的视角来看各层是如何配合处理一个请求的：

对于来自客户端的请求主要有两种：1. 对于页面的请求 2. 对于接口的请求。一个成功的请求应该是经过了以上各层的处理的

首先，一个请求来到服务端，会根据路径和方法等特征被路由交给对应的处理程序（即业务层中的 controller）来处理，在 controller 处理的前后，会被各种中间件包围，例如 错误捕获中间件、接口签名校验中间件、接口参数校验中间件。这种“包围”实际上是通过 koa 的“洋葱圈”模型实现的，即每个中间件实际上是一个 async 函数，每个中间件中 await next() 之前的代码逻辑的执行顺序与中间件注册顺序相同， await next() 之后的代码逻辑的执行顺序与中间件注册顺序相反，通过该特性实现接口请求的预处理与后置处理。controller 内部可能会调用多个 service 来实现业务逻辑处理，service 提供直接与数据层交互的原子化能力，controller 调用这些能力获取数据并组装后返回给客户端。 

以上能力是 elpis-core 在运行时提供的能力，实际上是将用户编写的逻辑代码通过各种 loader 挂载到 Koa 实例上（即加载到内存中）以达到运行时的效果，所以这就要求用户在编写逻辑代码时需要按照某种“约定”来安排目录结构，从而各种文件可以被对应的 loader 来处理并挂载。例如 router 定义应该放在 app/router 目录下，从而可以被引擎中的 router-loader 处理。

这种将文件通过 loader 加载到内存中的方式为 elpis-core 带来了强大的可扩展性，用户只需要在对应文件夹中编写业务代码，loa就会被对应 loader 加载，从而用户可以专注于业务逻辑实现，达到“声明即生效”的效果。
